# Deploy ARCH Slurm from a remote workstation (Ansible)

End-to-end guide for running the CLOACK Ansible playbooks **from a workstation
other than the ColdFront host** against a bare-metal Slurm cluster (controller
+ login + compute). Goal: reproduce the same binary stack the Docker dev
deploy produces — same Slurm version, same SPANK plugins, same UIDs 449
(munge) / 450 (slurm) — and validate the targets BEFORE touching them with
[slurm-verify/playbook.yml](../ansible/slurm-verify/playbook.yml).

Source of truth for the `ansible/` tree: the Python dict `STATIC_FILES` in
[coldfront/custom/plugins/arch_sync/ansible_scaffold_data.py](../coldfront/custom/plugins/arch_sync/ansible_scaffold_data.py).
The tree is **gitignored** — regenerated on demand via
`./start.sh slurm ansible all`. Edits to playbooks must be re-embedded with
[scripts/regen_ansible_scaffold.py](../scripts/regen_ansible_scaffold.py) so
fresh clones keep producing the same content (memory
`feedback_ansible_scaffold_refresh`).

## Audience

Operators with shell access to the **ColdFront host** (where the source of
truth lives) AND SSH key access to the **bare-metal compute nodes** (Rocky
Linux 9). The workstation is in the middle.

**Topology (skipjack, stage 3/4 — hybrid):** the controller core
(`slurmctld` / `slurmdbd` / `slurmrestd`) **and `login01`** run
**containerized** on the ColdFront host. `login01` stays containerized on
purpose — it carries the TOTP + JHED/Entra OTP-SSH stack (`pam_kc_ssh`,
memories `project_otp_ssh` / `project_auth_passwordless_model`) that the
legacy bare-metal `login01` does not have. Only the **compute** workers are
bare-metal and managed by Ansible (`slurm-client`).

> **Core container names depend on `prefix_core_containers`.** The `docker exec`
> commands below refer to the accounting daemons as `skipjack-slurmdbd` /
> `skipjack-slurmrestd`. Those per-cluster names exist **only when
> `SlurmCluster.prefix_core_containers=True`** (set so the container names match the
> site DNS `172.16.1.2 → skipjack-slurmctld skipjack-slurmdbd skipjack-slurmrestd`;
> see the [flag note in the 4-stage guide](./deploy-pipeline-4-stages.md)). With the
> flag **OFF** (the default) the shared core is `slurmdbd` / `slurmrestd` — substitute
> those names. `skipjack-slurmctld` (the controller the compute nodes register with)
> is always per-cluster regardless of the flag, and the MariaDB backend is always
> `slurm-db` (never prefixed).

```
┌──────────────────────────┐         ┌──────────────────────────┐
│  Workstation (you)       │  SSH    │  Bare-metal compute      │
│  - Python 3.10+          │ ──────▶ │   - b201, b202, …        │
│  - ansible-core ≥ 2.15   │         │     (slurmd only)        │
│  - ansible/ tree (rsync) │         └──────────────────────────┘
└──────────────────────────┘                    ▲ slurmd registers
            ▲                                    │ with containerized ctld
            │ rsync ansible/                     │
┌─────────────────────────────────────────────────────────┐
│  ColdFront host (containerized core + login)             │
│  - slurmctld / slurmdbd / slurmrestd                     │
│  - login01  (TOTP + JHED/Entra OTP-SSH)                  │
│  ./start.sh slurm ansible all   → generates ansible/     │
└─────────────────────────────────────────────────────────┘
```

> For a **fully** bare-metal cluster (controller + login on bare metal too),
> the `slurm-server` playbook and the controller inventory groups below still
> apply — populate `[skipjack_slurmctld/dbd/restd]` and run server → client.
> The skipjack stage 3/4 path keeps those containerized, so it runs
> **`slurm-client` only**.

> **Source-of-truth rule (`feedback_primary_repo_only`):** never edit the
> `ansible/` tree on the workstation as a permanent change. Edit
> `ansible_scaffold_data.py` on the ColdFront host, re-run
> `scripts/regen_ansible_scaffold.py`, commit, push, then rsync. The
> workstation is a *deployment vehicle*, not a development environment.

## Two deployment modes

There are two ways to run the same `slurm-client` / `slurm-server`
playbooks. Pick the one that fits the task:

| | **Mode A — remote workstation** (sections 1–9) | **Mode B — ColdFront UI button** ([section 10](#10-mode-b--one-click-deploy-from-the-coldfront-ui-per-node)) |
|---|---|---|
| Where Ansible runs | A separate workstation | Inside the `coldfront` container |
| Trigger | `ansible-playbook` CLI | **Deploy (Ansible)** button on the SlurmNode admin page |
| SSH user on target | `coldfront` (matches the inventory) | `coldfront` (hardcoded in the generated inventory) |
| SSH key | yours (`ssh-copy-id`) | auto-generated by the coldfront entrypoint |
| Scope | compute nodes (`slurm-client`); add server only for a full bare-metal cluster | one node at a time (`--limit <host>`), client only |
| Best for | first full bring-up of a cluster | adding / re-provisioning a single compute or login node once the core exists |

Mode B is the per-node path most operators use day-to-day; Mode A is the
full-cluster bring-up. Both consume the **same** `ansible/` tree and the
same source of truth.

below into the node's `coldfront` user. For the skipjack hybrid path the targets
are the **compute nodes only** — the controller and `login01` are containerized
on the ColdFront host and are not Ansible targets.

Once the key is authorized, verify passwordless SSH + `sudo` NOPASSWD from
wherever you run Ansible (here: `login03`), pointing `-i` at the **private**
key (`id_ed25519` — NOT the `.pub`):

```bash
# On login03. The deploy key must be readable here on the shared tree:
export ANSIBLE_PRIVATE_KEY_FILE=/home/jhu/admin/ansible/ansible-ssh/id_ed25519

ssh -i "$ANSIBLE_PRIVATE_KEY_FILE" coldfront@b201 'sudo -n true' \
  && echo "sudo NOPASSWD ok"
```

**Required env var for the run.** From login03 the playbooks read `CLUSTER_NAME`
off the control node to template `hosts:` (set once per shell):

```bash
export CLUSTER_NAME=skipjack                 # templates the playbook hosts: → skipjack_baremetal
```

The containerized core (`openldap`, `skipjack-slurmctld/dbd/restd`) is resolved
from the bare-metal nodes via **site DNS** — an A record per name pointing at the
ColdFront host's routable IP. CLOACK no longer injects `/etc/hosts` (a Docker
bridge IP like `172.22.x.x` is not routable off-host); make sure those names
resolve on each node (`getent hosts skipjack-slurmctld`) before running.

Set `CLUSTER_NAME` per-run with `-e cluster_name=<name>` instead if you prefer;
the private key likewise via `--private-key <path>`. Either way the inventory
stays generator-owned — no need to edit `clusters/<name>/inventory.ini`.

## 2. Get the `ansible/` tree onto the workstation

The tree is generated on the ColdFront host and copied over. Two paths —
pick whichever fits your workflow.

### Path A — generate on the ColdFront host (preferred)

On the **ColdFront host**:

```bash
cd /opt/cloack                # repo root (standard install path)
./start.sh slurm ansible all  # writes ./ansible/*
```

Collections are installed by the §1 step (`ansible-galaxy collection install
-r ansible/requirements.yml`) — `ml load ansible-core` provides ansible-core
but **not** the `community.general` / `ansible.posix` collections.

### Path B — generate on the workstation directly

Only viable if you have a checkout of the full repo on the workstation
(rare for ops staff). The generator is just a Python dict serialisation:

```bash
cd /opt/cloack
python3 -c "
from coldfront.custom.plugins.arch_sync import ansible_scaffold_data as m
import os, sys
for path, content in m.STATIC_FILES.items():
    full = os.path.join('ansible', path)
    os.makedirs(os.path.dirname(full), exist_ok=True)
    with open(full, 'w') as f: f.write(content)
print(f'wrote {len(m.STATIC_FILES)} files')
"
```

Either path leaves you with a tree that matches the [layout in
ansible/README.md](../ansible/README.md):

```
ansible/
├── ansible.cfg
├── requirements.yml
├── group_vars/all.yml             ← STUB (not auto-loaded — see note below)
├── clusters/<name>/               ← per-cluster inventory + slurm.conf
│   ├── inventory.ini
│   ├── slurm.conf
│   ├── group_vars/all.yml         ← per-cluster overrides (cluster_name, …)
│   └── README.md
├── roles/common/defaults/main.yml ← shared defaults (versions, UIDs, sssd, …)
├── roles/{common,slurm_users,slurm_build,munge,slurm_config,slurm_server,slurm_client}/
├── slurm-verify/playbook.yml      ← read-only pre-flight (you start here)
├── slurm-server/playbook.yml      ← controller + dbd + restd
└── slurm-client/playbook.yml      ← compute + login
```

## 3. Fill in `clusters/<name>/inventory.ini`

The generator emits a placeholder inventory. Edit it with the real
hostnames/IPs for your cluster. Required groups (drop the `<cluster>_`
prefix if your inventory is single-cluster):

```ini
# clusters/skipjack/inventory.ini
#
# Hybrid topology (skipjack stage 3/4): the controller core
# (slurmctld/slurmdbd/slurmrestd) AND login01 run *containerized* on the
# ColdFront host — so [skipjack_slurmctld/dbd/restd] stay empty and login01
# is NOT listed here. login01 stays containerized to keep its TOTP + JHED/
# Entra OTP-SSH stack (the bare-metal login01 at 172.16.2.11 lacks it).
# Only the bare-metal COMPUTE nodes are Ansible targets (slurm-client).
# Values below are stage 4 (portal / prod) on the mgmt network
# (*.mgmt.ai.cluster, 172.16.18.x).
#
# Stage 3 (edulogin pre-prod) usually has a single bare-metal compute node:
#   c01 ansible_host=172.22.10.1 ansible_user=coldfront node_type=compute

[skipjack_slurmctld]
# (containerized on the ColdFront host — leave empty; see note above)

[skipjack_slurmdbd]
# (containerized on the ColdFront host — leave empty)

[skipjack_slurmrestd]
# (containerized on the ColdFront host — leave empty)

[skipjack_baremetal]
b201 ansible_host=172.16.18.1 ansible_user=coldfront node_type=compute
b202 ansible_host=172.16.18.2 ansible_user=coldfront node_type=compute
# gpu-01 ansible_host=172.16.18.20 ansible_user=coldfront node_type=compute gres="gpu:4"

[all:vars]
cluster_name=skipjack
slurm_conf_src=ansible/clusters/skipjack/slurm.conf
munge_key_src=../../slurm/conf/munge.key
ansible_user=coldfront
ansible_become=true
ansible_become_method=sudo
```

> **Keep the `[all:vars]` block** — the generator (`export_baremetal_inventory`)
> always emits it, and `cluster_name` is **mandatory**: every playbook templates
> its `hosts:` from `{{ cluster_name }}` (e.g. `{{ cluster_name }}_slurmctld`).
> Drop it and `ansible-playbook` aborts before running anything with
> `'cluster_name' is undefined`. If you ever hand-build an inventory without it,
> pass `-e cluster_name=skipjack` on the command line instead.

Verify connectivity before doing anything else:

```bash
cd ~/ansible
ansible -i clusters/skipjack/inventory.ini all -m ping
```

Every host must return `pong`. Fix DNS, SSH keys, or `become_pass` issues
before proceeding.

`clusters/<name>/group_vars/all.yml` carries per-cluster overrides; the
shared defaults (Slurm version, paths, munge/slurm UIDs, sssd) live in
`ansible/roles/common/defaults/main.yml`. They are role defaults — **not**
tree-root `group_vars/all.yml`, which Ansible does not auto-load for the
standard `-i clusters/<name>/inventory.ini <play>/playbook.yml` invocation
(that file is a stub). Role defaults are lowest precedence, so anything you put
in `clusters/<name>/group_vars/all.yml` overrides them. Don't touch UIDs 449 /
450 — they MUST match the LDAP records ColdFront writes (memory
`project_uid_gid_ranges`).

> **LDAP/SSSD on bare metal:** the `sssd_ldap_*` defaults are **dev/docker**
> values (`sssd_ldap_uri: ldap://openldap`) and will not resolve from a
> bare-metal node. For each real cluster, override them in
> `clusters/<name>/group_vars/all.yml` with the LDAP server the compute nodes
> can actually reach (e.g. `sssd_ldap_uri: ldap://ldap.arch.jhu.edu`,
> `sssd_ldap_base`, `sssd_ldap_bind_dn`), and vault the bind password in
> `clusters/<name>/group_vars/secrets.yml`. SSSD must resolve cluster users so
> jobs run under the right UID.

## 4. Pre-flight: `slurm-verify` (read-only)

```bash
ansible-playbook -i clusters/skipjack/inventory.ini slurm-verify/playbook.yml
```

`slurm-verify` makes **no changes**. It checks:

| Gate | What it verifies |
|---|---|
| Control-node Python | the interpreter running `ansible-playbook` is ≥ 3.10 (hard fail with a venv-rebuild hint — runs on `localhost`, before any target is touched) |
| OS family | Rocky Linux 9.x (or RHEL-compatible) |
| Time sync | chronyd present and synced |
| Munge UID | 449 system user exists |
| Slurm UID | 450 system user exists |
| Munge key | present and `0400 munge:munge` |
| Slurm version | matches `slurm_version` in group_vars (default `25.11.2`) — WARN if absent (will install) |
| SPANK | C headers present, `plugstack.conf` resolvable, `.so` linker symbols available |
| Firewall | ports 6817 (ctld) / 6818 (slurmd) / 6819 (dbd) / 6820 (restd) open per-group |
| SSSD entry cache | `entry_cache_timeout`, `entry_cache_user_timeout`, `entry_cache_group_timeout` ≤ 60 on login nodes (WARN otherwise — per memory `project_sssd_flush_hook`) |

A non-green play recap means **stop here**. Fix the OS/network problem on
the target host before running `slurm-server` or `slurm-client` — those
playbooks WILL make changes and won't roll back cleanly from a broken base.

## 5. Optional: `prod-sim` dry-run (`--check --diff`)

A wrapper on the mgm02 (ColdFront) host runs the **`slurm-verify`** playbook in
Ansible's `--check --diff` mode against the per-cluster inventory (no changes
applied). It is the §4 pre-flight with `--check --diff` added — a read-only
sanity pass, **not** a dry-run of the provisioning playbooks:

```bash
# On the mgm02 host. Needs ansible-playbook on PATH +
# ansible/clusters/<name>/inventory.ini (regenerate with
# `./start.sh slurm ansible <name>`). Read-only — no changes applied.
./start.sh slurm prod-sim skipjack
```

The workstation equivalent is the same verify playbook with the same flags:

```bash
ansible-playbook -i clusters/skipjack/inventory.ini slurm-verify/playbook.yml --check --diff
```

To preview the **actual provisioning** (every file rewrite + service restart
the real run would apply — useful on a host that already has Slurm you don't
want to clobber), run the provisioning playbook itself in check mode as a
**separate** step. For the skipjack hybrid path the controller core is
containerized, so only the compute play applies:

```bash
# Full bare-metal cluster only — uncomment to also dry-run the controller:
# ansible-playbook -i clusters/skipjack/inventory.ini slurm-server/playbook.yml --check --diff
ansible-playbook -i clusters/skipjack/inventory.ini slurm-client/playbook.yml --check --diff
```

Use this BEFORE the real provision — particularly on hosts that already
have a Slurm install you don't want to clobber. The `--check` mode shows
every file rewrite and service restart the playbook would do.

## 6. Real provision

### 6a. Controller / DBD / RESTD — skipped for the hybrid path

For **skipjack (stage 3/4)** the controller core *and* `login01` run
containerized on the ColdFront host, so there is **no `slurm-server` run** —
the `slurmctld`/`slurmdbd`/`slurmrestd` containers are already up. Confirm
the live controller before provisioning compute (run on the ColdFront host):

```bash
docker exec skipjack-slurmctld scontrol ping       # → Slurmctld(primary) UP
docker exec skipjack-slurmdbd  sacctmgr show cluster -n -P | grep skipjack
```

> **Full bare-metal cluster only** (controller on bare metal): run
> **server first, then client** — `slurmd` won't register without a live
> `slurmctld`, and `slurmctld` won't start without `slurmdbd`. After
> populating `[skipjack_slurmctld/dbd/restd]`:
> `ansible-playbook -i clusters/skipjack/inventory.ini slurm-server/playbook.yml`
> (`common` → `slurm_users` UID 449/450 → `slurm_build` 25.11.2 →
> `munge` → `slurm_config` + SPANK → `slurm_server` systemd units), then
> wait for `ssh coldfront@<ctld> 'systemctl is-active slurmctld'` → `active`.

### 6b. Compute nodes (bare-metal)

```bash
ansible-playbook -i clusters/skipjack/inventory.ini slurm-client/playbook.yml
```

`common` / `slurm_users` / `slurm_build` / `munge` / `slurm_config`
foundation, plus:

7. `slurm_client` → `slurmd` systemd unit, `cgroup.conf`, optional `gres.conf` if the host has `gres=` in inventory

(`login01` is containerized, so it is **not** in this run — no bare-metal
login node to provision on the skipjack path.) After this finishes, the new
compute nodes should appear in `sinfo` — query the containerized controller
on the ColdFront host:

```bash
docker exec skipjack-slurmctld sinfo -h
# skipjack*  up  infinite  1  idle  b201
# skipjack*  up  infinite  1  idle  b202
```

### 6c. Configless mode (opt-in)

When `SlurmCluster.configless=True` (default off, activated by
`SlurmctldParameters=enable_configless` in `slurm.conf`), the containerized
controller (`skipjack-slurmctld`) is the **single source of config** and
**serves** the `.conf` set over port **6817** — `slurm.conf` (`:ro`, the
operator's static file is truth), `gres.conf`, `cgroup.conf`, and
`plugstack.conf` (all per-cluster, co-located under
`${BASE_DATA}/slurm/<cluster>/`). Bare-metal nodes fetch config on boot instead
of carrying a local copy:

```bash
# compute node (slurmd fetches slurm.conf/gres/cgroup — no local /etc/slurm/slurm.conf)
slurmd --conf-server <coldfront-host>:6817
# login node (sackd, Slurm 23.11+: gives srun/sacct/sinfo config with no local file)
sackd --conf-server <coldfront-host>:6817
```

**What configless serves vs. what stays local:** configless serves `.conf`
files **only, never binaries**. So each node still needs, installed locally by
the `slurm-client` role (or baked into the image):

- the client `.lua` (`cli_filter.lua` / `rates.lua` / `qos_config.lua`),
- the `prolog.d` / `epilog.d` scripts,
- the SPANK `.so` objects in `/usr/lib64/slurm/`. `plugstack.conf` **is** served,
  but it only *names* the `.so`; the file itself must exist on the node. Its
  entries (`myspank.so`, `spank_reemit_cost.so`, `spank_pyxis.so`) are all
  `optional` = **fail-soft** (log the failure, don't abort the job), so a served
  `plugstack` won't break job launch even if a node is missing a `.so`.

> **Port must be 6817, not `6817+(pk-1)`.** With `prefix_core_containers` the
> controller is published on host port **6817** flat. An older pk-offset (e.g.
> `6818` for pk=2) is unreachable by bare-metal `slurmd`/`sackd` fixed on 6817 →
> nodes stuck `Down` / configless "connect failure".

> **Per-cluster mount files must exist as FILES first.** `slurm.conf`,
> `gres.conf`, `cgroup.conf`, and `plugstack.conf` must all exist under
> `${BASE_DATA}/slurm/<cluster>/` (e.g. `/opt/mprov/cloack/var/slurm/skipjack/`)
> **before** `docker compose up -d`. If a mount source is missing, Docker
> silently creates a phantom empty **directory** at that path and slurmctld
> fails reading it as a dir. Stage any missing one:
> `cp slurm/conf/<file> ${BASE_DATA}/slurm/skipjack/<file>`.

**Node bring-up after a controller recreate.** Nodes show `unk` (UNKNOWN)
briefly while configless `slurmd` re-registers — a normal blip; wait ~1 min and
re-check `sinfo -N`. If the controller log warns *"Node `<X>` appears to have a
different slurm.conf than the slurmctld"* (CONF_HASH mismatch), that node has a
stale cached/local `slurm.conf` — force it to re-fetch, don't paper over it with
`DebugFlags=NO_CONF_HASH`:

```bash
ssh coldfront@b201 'sudo systemctl restart slurmd'        # re-fetch via --conf-server
docker exec skipjack-slurmctld scontrol update NodeName=b201 State=RESUME
```

## 7. Post-deploy validation

### Functional smoke test

Submit from the containerized `login01`; the job should land on a freshly
provisioned bare-metal node (`b201`/`b202`). Run on the ColdFront host:

```bash
docker exec skipjack-login01   sudo -u <a_test_user> sbatch --wrap="hostname && date"
docker exec skipjack-slurmctld squeue   # job appears, runs on b201/b202, then clears
docker exec skipjack-slurmctld sacct -X --starttime=$(date -d "5 min ago" +%FT%T)
```

### LDAP resolution via SSSD (on the new bare-metal node)

This is what the `slurm-client` run just configured — verify SSSD resolves
LDAP users directly on the compute node:

```bash
ssh coldfront@b201 'getent passwd <a_test_user>'   # returns the LDAP entry
ssh coldfront@b201 'id <a_test_user>'              # shows correct groups
```

### Admin escalation for `cn=admin` members

Members of LDAP `cn=admin` must be able to `sudo` on the bare-metal nodes —
that's how operators log in interactively. Verify on the new node:

```bash
ssh coldfront@b201 'getent group admin'        # LDAP group present
ssh coldfront@b201 'sudo -l -U <admin_user>'   # NOPASSWD: ALL
```

> **Pending dependency** — see [related plan](./README.md): the
> `/etc/sudoers.d/admin` file with `%admin NOPASSWD: ALL` is currently
> created by the Docker entrypoint
> ([slurm/slurmd/entrypoint.sh:44-46](../slurm/slurmd/entrypoint.sh)) but
> is NOT yet in the Ansible `slurm_client` role. If `sudo -l` shows the
> admin user has no rules, that's the gap — it must be added to
> `ansible_scaffold_data.py` key `roles/slurm_client/tasks/main.yml` for
> bare-metal parity. Re-run `scripts/regen_ansible_scaffold.py` after the
> edit, commit, push, rsync to the workstation, and re-run
> `slurm-client/playbook.yml --tags sudoers`.

### Cross-cluster federation check (multi-cluster only)

If this is a federated deploy where `slurmdbd` is shared:

```bash
docker exec skipjack-slurmctld sacctmgr show cluster format=Cluster,Federation -P
```

All federated clusters should appear here. ColdFront's `SlurmCluster.name`
must match exactly (case-sensitive).

### Register the nodes in ColdFront (`import_slurm_cluster`)

ColdFront is the source of truth, but a bare-metal node provisioned by
Ansible isn't known to ColdFront until you snapshot it back. The
`import_slurm_cluster` management command reads **nodes / partitions / QOS**
from the live cluster (via `slurmrestd`) into ColdFront so the new `b201`/
`b202` appear in Service Health, billing, and the cluster admin pages
(memory `project_import_slurm_cluster`). Run it on the ColdFront host:

```bash
# dry-run — show what would be created/updated, change nothing
docker exec coldfront coldfront import_slurm_cluster --cluster skipjack --check

# create any missing SlurmNode / SlurmPartition / SlurmQOS rows (idempotent)
docker exec coldfront coldfront import_slurm_cluster --cluster skipjack

# also refresh whitelisted Slurm-owned fields (cpus, real_memory, gres, state,
# ip_address, partition times) on existing rows — ColdFront-owned billing/UI
# fields are left untouched, so this is safe to run on a schedule
docker exec coldfront coldfront import_slurm_cluster --cluster skipjack --update
```

The `SlurmCluster` record itself must already exist (created via
`./start.sh slurm create skipjack` or Django Admin → Slurm Clusters); the
import populates its nodes/partitions, it does not create the cluster.

Beyond the CLI, the same engine
([cluster_manager.import_from_cluster()](../coldfront/custom/plugins/arch_sync/cluster_manager.py))
is exposed two more ways:

- **Django Admin (no shell needed)** — Slurm Clusters changelist → select rows →
  Action dropdown ([admin.py](../coldfront/custom/plugins/arch_sync/admin.py)):
  ⬇ *Import (create only)*, ⟳ *Update (overwrite Slurm-owned fields)*, and
  🔍 *Preview drift (`--update --check`)* which lists up to 5 `field: old→new`
  diffs inline.
- **Scheduled (automatic)** — the `sync_bare_metal_topology` django-q task
  ([tasks.py](../coldfront/custom/plugins/arch_sync/tasks.py)) runs `--update`
  for every active **bare-metal** cluster every 10 min
  ([base.py](../coldfront/custom/config/base.py)), so a node added by Ansible /
  `sacctmgr` surfaces in ColdFront with no operator action. It **skips container
  clusters** — their topology flows the other way (UI → DB → `slurm.conf`), so
  an import would overwrite UI-authored values.

The node **`ip_address`** (NodeAddr) round-trips too, but only when slurmrestd
reports a *real IP* — Slurm defaults NodeAddr to the node *name*, so a
hostname-valued address is ignored and a manually-pinned UI value (the one the
[Deploy button](#10-mode-b--one-click-deploy-from-the-coldfront-ui-per-node)
uses) is preserved. `--update` never clears a field REST omitted, so a partial
response can't wipe data.

## 8. Editing playbooks — round-trip workflow

If during validation you discover a missing task (e.g. the sudoers gap
above), the source of truth is **never** the workstation's
`ansible/` tree:

1. SSH into the ColdFront host
2. Edit `ansible/<role>/tasks/main.yml` directly to validate the change
3. Re-run the affected playbook to confirm the fix works on a target host
4. **Persist the change**: edit the corresponding string inside
   `coldfront/custom/plugins/arch_sync/ansible_scaffold_data.py`
   (`STATIC_FILES[<path>]`)
5. Run `python3 scripts/regen_ansible_scaffold.py` to verify the embedded
   content matches the on-disk file (idempotent — should report zero diff
   if step 4 was complete)
6. Commit + push
7. On the workstation: `rsync` the fresh tree (path A in section 2 above)

Skipping step 4 means the next operator who regenerates from
`ansible_scaffold_data.py` will see their changes overwritten — memory
`feedback_ansible_scaffold_refresh` documents this trap.

## 9. Cleanup / rollback

`slurm-verify` is read-only, so nothing to clean.

`slurm-client` installs + enables the `slurmd` systemd unit on each
bare-metal compute node; to revert manually (per node):

```bash
ssh coldfront@b201 'sudo systemctl stop slurmd && sudo systemctl disable slurmd'
ssh coldfront@b202 'sudo systemctl stop slurmd && sudo systemctl disable slurmd'
# Binaries stay in /usr/local/sbin (slurm_build does `make install`).
# If you need them gone: rm -rf /usr/local/sbin/slurm* /etc/slurm/* /var/spool/slurm*
```

The containerized controller + `login01` are managed by Docker, not Ansible
— stop them with `./start.sh slurm stop` / `docker stop skipjack-slurmctld
skipjack-login01`, never via systemd. For a **full bare-metal cluster** the
controller teardown is `ssh coldfront@<ctld> 'sudo systemctl stop slurmctld
slurmdbd slurmrestd'`.

There's no `--undo` playbook today; if rollback becomes a recurring need,
that's a follow-up scaffolding addition.

## 10. Mode B — one-click deploy from the ColdFront UI (per node)

Instead of running playbooks from a workstation, ColdFront can run
[ansible/slurm-client/playbook.yml](../ansible/slurm-client/playbook.yml)
against a **single** bare-metal node directly from the admin UI. On the
SlurmNode change page (only for `deploy_type=bare_metal`), the **Deploy
(Ansible)** button calls `cluster_manager.ansible_deploy_node()`
([coldfront/custom/plugins/arch_sync/cluster_manager.py](../coldfront/custom/plugins/arch_sync/cluster_manager.py)),
which regenerates the per-cluster bundle and runs:

```bash
ansible-playbook -i ansible/clusters/<cluster>/inventory.ini \
  ansible/slurm-client/playbook.yml --limit <hostname> \
  -u coldfront --become --become-method sudo
```

Ansible runs **inside the `coldfront` container** (as root, `HOME=/root`),
connecting over SSH to the `coldfront` user on the target. The SSH key is
auto-generated and persistent (see below) — no `ssh-copy-id` from a
workstation. Use this to add or re-provision one node at a time once the
cluster core (`slurmctld`/`slurmdbd`/`slurmrestd`) already exists.

> **Saving a node never deploys.** Filling the form and clicking **SAVE**
> only writes the DB row and regenerates `inventory.ini`. The actual push is
> the separate **Deploy (Ansible)** button. The `ip_address` field changes
> *which address* the deploy connects to, not *whether/when* it runs.

### Once only (per stack)

**0. Activate the image.** The `coldfront` image bakes in `ansible-core`,
`openssh-client`, and `sshpass`, and the entrypoint auto-generates the deploy
key. Rebuild + recreate only the coldfront container:

```bash
docker compose -f docker-compose-dev.yml up -d --build coldfront    # dev
# confirm the tools + key are present:
docker exec coldfront sh -c 'ansible-playbook --version | head -1; command -v ssh-keygen'
docker logs coldfront 2>&1 | grep -i "ansible deploy key"
```

The SSH key persists across recreates: dev uses the named volume
`arch_coldfront_ssh` → `/root/.ssh`; prod uses the bind mount
`${BASE_ETC}/coldfront/ansible-ssh:/root/.ssh`.

**1. Export the munge key** so the bare-metal node shares the cluster's auth
secret (most common cause of a node that installs but never registers). The
munge role copies the key from `munge_key_src`. Stage it on the shared tree
`/home/jhu/admin/ansible/` (visible on all nodes) and point the var at it with
an **absolute** path — the default `../../slurm/conf/munge.key` only resolves
correctly when the tree lives under `<repo>/ansible/`:

```bash
# Stage the SAME key the containers use (slurmctld + coldfront both mount the
# munge-key volume at /etc/munge). Run where the container lives (mgm02):
mkdir -p /home/jhu/admin/ansible/munge
docker cp skipjack-slurmctld:/etc/munge/munge.key /home/jhu/admin/ansible/munge/munge.key
chmod 400 /home/jhu/admin/ansible/munge/munge.key
```

Point the munge role at it as a per-cluster override (not a hand-edit of the
generator-owned `inventory.ini`) — `-e` per run, or persisted in group_vars:

```bash
ansible-playbook ... -e munge_key_src=/home/jhu/admin/ansible/munge/munge.key
# or persist: clusters/skipjack/group_vars/all.yml
#   munge_key_src: /home/jhu/admin/ansible/munge/munge.key
```

`munge.key` is a secret — the `ansible/` tree is gitignored, never commit it. It
MUST match the cluster's key exactly (don't generate a fresh one), or the node's
munge can't authenticate with the containerized `slurmctld`/`slurmdbd`.

**2. Generate the Ansible scaffold** (playbooks + roles):

```bash
./start.sh slurm ansible all
```

### Per node (each new bare-metal host — example c01 / 172.22.10.1)

**A. Prepare the target host** (run as root on c01). The `coldfront` account
is a **local service user**, NOT a cluster user — keep it out of the
affiliation home tree (`/home/jhu`, `/home/schmidt`):

```bash
# create if not exist
# useradd --system --create-home --home-dir /var/lib/coldfront --shell /bin/bash coldfront
echo 'coldfront ALL=(ALL) NOPASSWD:ALL' > /etc/sudoers.d/coldfront
chmod 440 /etc/sudoers.d/coldfront && visudo -c

# authorize the public key from step 3
install -d -m 700 -o coldfront -g coldfront /var/lib/coldfront/.ssh

cp /home/jhu/admin/ansible/ansible-ssh/id_ed25519.pub /var/lib/coldfront/.ssh/authorized_keys 

chown coldfront:coldfront /var/lib/coldfront/.ssh/authorized_keys
chmod 600 /var/lib/coldfront/.ssh/authorized_keys

# Rocky 9 SELinux: authorized_keys outside /home needs the ssh_home_t label,
# otherwise sshd silently refuses the key
# sudo dnf install policycoreutils-python-utils
semanage fcontext -a -t ssh_home_t '/var/lib/coldfront/\.ssh(/.*)?'
restorecon -Rv /var/lib/coldfront/.ssh
```

Verify connectivity from the container (must print `uid=0(root)`):

```bash
docker exec coldfront ssh -o StrictHostKeyChecking=accept-new \
  coldfront@172.22.10.1 'sudo -n id'
```

**B. Register the node in ColdFront** — Admin → Slurm Nodes → **Add Slurm
Node**:

- Cluster: e.g. `skipjack` · Hostname: `c01` · **IP address: `172.22.10.1`**
- Deploy type: **Bare-metal** · Node type: Compute · set Cpus / Real memory / Gres
- **SAVE** → regenerates `ansible/clusters/<cluster>/inventory.ini` with
  `c01 ansible_host=172.22.10.1`. The IP also becomes `NodeAddr=172.22.10.1`
  in the generated `slurm.conf`.

The IP field is what makes Mode B work when the hostname does not resolve
from inside the coldfront container — leave it blank only if DNS/SSSD
already resolves the hostname.

**C. Deploy** — open the saved `c01` node → **Deploy (Ansible)** → confirm.
The playbook installs slurmd + munge + SSSD + enroot + ARCH CLI tools +
prolog/epilog and copies `slurm.conf`. The last lines of output appear as a
success/error message on the admin page.

**D. Make the node join the scheduler + verify.**

> ⚠️ **Topology note.** If the cluster's core is containerized
> (`cluster_type=container`, e.g. dev `arch-dev`), a bare-metal node joining
> it is the hybrid "Model D" layout. Step C **installs** slurmd, but for the
> node to register it must (1) reach the controller ports — `slurmctld` 6817,
> `slurmdbd` 6819 — across the container/host boundary, and (2) be listed as
> `NodeName=c01 NodeAddr=...` in the controller's `slurm.conf`. For a clean
> bare-metal test, prefer a dedicated bare-metal cluster
> (`cluster_type=bare_metal`, `core_services=dedicated`) and bring its core
> up with Mode A section 6a first.

```bash
# regenerate the controller's slurm.conf with the new node, then reload
./start.sh slurm restart
docker exec skipjack-slurmctld scontrol reconfigure
# verify
docker exec skipjack-slurmctld sinfo -N
docker exec skipjack-slurmctld scontrol show node c01
```
