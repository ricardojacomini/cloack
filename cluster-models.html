When isolation is needed
Model A — Single Cluster
Benefits
  • Simple to operate — one slurmctld to manage
  • Unified scheduling — scheduler sees all resources
  • One slurm.conf, one set of partitions
  • Scales comfortably to ~10,000 nodes
Drawbacks
  • Single point of failure — slurmctld crash affects all
  • Maintenance downtime impacts every user
  • All hardware shares the same scheduling policies
Model B — Federation
Benefits
  • Fault isolation — one cluster crash doesn’t affect others
  • Independent maintenance — upgrade GPU without touching CPU
  • Optimized scheduling policies per hardware type
  • Dev cluster fully isolated from production
  • Unified billing via shared slurmdbd
  • Cross-cluster submission: sbatch -M schmidt
Drawbacks
  • More complex to operate (multiple slurmctld)
  • Separate slurm.conf per cluster
  • Resource fragmentation between clusters

Recommendation for JHU: Start with Model A (single cluster with partitions). Migrate to Model B when hardware diversity or operational isolation justifies the complexity. The ColdFront SlurmCluster model supports both — migration is infrastructure-only, no code changes needed.
