scavenger Partition
  • Default partition for all PIs
  • Preemptible — jobs can be interrupted by premium
  • Never billed (QOS: scavenger, is_free=True)
  • Default allocation uses this partition
  • ssci-* accounts also always free
premium Partition — PAID
  • High-priority, non-preemptible
  • Full billing rate ($0.35/SU CPU, $1.20/GPU-hour)
  • Requires paid allocation (project with abbreviation + IO number)
  • Weekly spend cap support via GrpTRESMins=billing=
  • QOS: premium (priority=20) or normal (priority=0)
gpu-a100 Partition — PAID
  • GPU workloads (A100, H100)
  • TRESBillingWeights: CPU=1.0,GRES/gpu=2.0
  • Billing rate per GPU-hour
  • Requires TRES types: cpu,gres/gpu,billing
dev Partition
  • Development and testing
  • Short time limits
  • Free (scavenger QOS)
  • Isolated from production workloads
sync_ldap_all — Daily 2:00 AM
  • Syncs all LDAP users (ou=jhu + ou=schmidt) into Django User table
  • Updates first/last name, email from LDAP attributes
  • Creates home directories for new users
  • Creates PI groups in LDAP for new PIs

coldfront sync_ldap

sync_slurm_all — Daily 3:00 AM
  • Bidirectional sync: ColdFront allocations ↔ Slurm accounts
  • Creates/updates account hierarchy (root → school → dept → PI)
  • Sets QOS, fairshare, GrpTRESMins per account
  • Skips users who haven’t accepted Terms of Service
  • Removes system accounts from associations
  • Updates Core Usage gauge from SlurmJob table

coldfront sync_slurm

collect_sacct — Every 15 min
  • Imports completed jobs from sacct into SlurmJob table
  • Maps jobs to allocations via slurm_account_name
  • Computes CPU/GPU/node time hours per job
  • Uses -M cluster_name for Federation support
  • Upsert (update_or_create) — safe to re-run

coldfront collect_sacct --current-week

sync_weekly_cap_usage — Every 5 min
  • Polls Slurm sreport for billing-TRES usage per account
  • Converts minutes to dollars and stores in AllocationAttribute
  • Feeds the Weekly Cap Usage ($) gauge on allocation detail page
  • Only runs for allocations with Weekly Cap ($) or Weekly Cap (Hours) set
check_and_run_core_hours_reset — Configurable
  • Checks if reset is due based on CoreHoursResetSettings period
  • Supports 6 periods: weekly, monthly, quarterly, semester, 4-month, annual
  • Per-allocation override via Next Reset attribute
  • Zeroes Core Usage (Hours) and per-user snapshots
  • Creates audit log entry
cleanup_task_queue — Hourly :30
  • Purges stuck and old Django-Q tasks
  • Removes completed tasks older than 24 hours
  • Removes failed tasks older than 7 days
  • Keeps the task queue lean for performance
Billing System
Two-Track Model
  • Pay Per Use — full rate (compute + O&M bundled)
  • Hardware Credit Holder — O&M only (credits offset compute)
Rate Catalog (BillingRate)
  • cpu_full / cpu_om — per SU
  • gpu_full / gpu_om — per GPU-hour
  • storage / backup — per TB/month
  • Versioned by effective_from / effective_to
Invoices
  • Per-PI monthly usage reports (Staff → Billing Report)
  • PDF download with fpdf2
  • PI self-service: My Usage & Billing page
Storage & Backup
  • storage_enabled / storage_tb — per-project storage allocation
  • backup_enabled / backup_tb — per-project backup allocation
  • Set at project creation/update by PI
  • Propagated to AllocationBilling on approval
  • Disabled for default projects
  • Shown on Project Detail with badges
  • Billed via storage and backup rate types
Job Metrics Dashboard (XDMoD-style)
  • URL: /jobs/dashboard/
  • Tabs: User, Account, Partition, QOS, State, Department, School
  • Date range picker + rows filter
  • Summary cards: CPU Hours, Wall Hours, Jobs, Users, Avg Wait
  • Expandable per-user job tables with details
  • CSV export
  • Data source: SlurmJob table (populated by collect_sacct)
Job Submit UI
  • URL: /jobs/submit/
  • Script editor + partition/account/time selectors
  • Live cost estimation from BillingRate catalog
  • Weekly cap budget display
  • ToS enforcement per account (PI must accept)
Terms of Service
  • Versioned ToS documents with required sections
  • Only one active version at a time
  • PI must accept before creating projects
  • Users without ToS acceptance are skipped by slurm_sync — no Slurm account created
  • Enforced on: project creation, allocation request, job submission UI
  • Acceptance tracked: user, timestamp, IP, context (project_create / allocation_request / reaccept)
  • Import via CSV: Admin Tools → Import ToS
Weekly Spend Cap
  • Set via Weekly Cap ($) or Weekly Cap (Hours) attribute
  • Dollar cap takes precedence
  • Conversion: $X × 1,000 × 60 = GrpTRESMins
  • Applied on account association (not per-QOS)
  • Detection: billing != cpu in GrpTRESMins
  • Spend gauge on allocation detail page (5-min sync)
  • sbalance --week CLI report
  • cli_filter.lua shows cap at job submit
