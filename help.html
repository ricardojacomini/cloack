Help & Reference Guide

Guide for PIs and users — billing, CLI tools, job submission, and more.

  1. PI creates project — sets title, abbreviation, department, IO number, storage/backup needs. Must accept ToS if active.
  2. PI requests allocation — chooses cluster resource, enters core hours.
  3. Staff reviews & approves — allocation moves to Active.
  4. Auto-seed on activation:
    • AllocationBilling created (copies IO, department, storage/backup from project)
    • Default attributes seeded: Core Usage, Extra Core Hours, Next Reset
  5. Slurm sync triggers — creates Slurm account, sets hierarchy, QOS, GrpTRESMins. Users without ToS acceptance are skipped.
  6. LDAP sync — ensures user exists in LDAP with home directory.
  7. Users submit jobs — billing tracked via sacct collection every 15 minutes.

Allocation Usage Gauges

Each allocation has its own Core Usage gauge. The data source depends on the allocation type:

GaugeAllocation TypeData SourceWhat It Shows
Core Usage (Hours) Default (scavenger) slurm2coldfront() → SlurmJob Free core-hours used (scavenger QOS)
Core Usage (Hours) Paid (custom project) slurm2coldfront() → SlurmJob Paid core-hours used (normal/premium QOS)
Weekly Cap Usage ($) Paid with cap set sync_weekly_cap_usage() → sreport Weekly dollar spend vs cap
  • Default allocations (project = PI username, no abbreviation) get scavenger QOS and are never billed.
  • Paid allocations (custom project with abbreviation) get normal/premium QOS and are billed.
  • Each allocation has its own Slurm account (pi_username vs pi_username_proj1), so jobs never mix between gauges.
  • The Weekly Cap gauge only appears when Weekly Cap ($) or Weekly Cap (Hours) is set on the allocation.
Hardware Credit Pools

Hardware purchases are converted to compute credits via HardwareCreditPool. Credits offset compute charges only, not O&M. Each pool has a granted date, expiry date, and tracks credits used/remaining.

Billing Enforcement in Slurm

How ColdFront billing settings translate to Slurm enforcement via coldfront2slurm() (runs in sync_slurm scheduled task).

1. Account Hierarchy — Who Can Submit

slurm.conf enforces: AccountingStorageEnforce=associations,limits,qos. Users without a Slurm account association cannot submit jobs. 

root (jhu)
  └─ it (school)        ← fairshare from School.shares
        └─ arch (department)    ← fairshare from Department.dept_shares
              └─ jsmith (PI allocation account)
                    └─ user1, user2  ← associations
User State in ColdFrontSlurm ActionEffect
Active on allocationMaxJobs=-1 (unlimited)Can submit jobs
Removed / inactiveMaxJobs=0Cannot submit; sacct history preserved
ToS not acceptedSkipped entirelyNo Slurm association created
2. GrpTRESMins — Usage Limits

Each allocation account gets a GrpTRESMins limit derived from the Core Hours allocation attribute. The TRES types are configured per cluster via SlurmCluster.tres_types (default: cpu,billing).

Scenariocpu=billing=
No weekly cap hours × 60 hours × 60 (same value)
Weekly cap set ($X) hours × 60 (core hours) $X × 1,000 × 60 (cap in billing minutes)

Detection: when billing ≠ cpu in GrpTRESMins, a weekly cap is active.

# Example: 1000 core hours + $50 weekly cap
sacctmgr update account set grptresmins=cpu=60000,billing=3000000 where account=jsmith_proj1

# No weekly cap: billing = cpu
sacctmgr update account set grptresmins=cpu=60000,billing=60000 where account=jsmith_proj1
3. QOS — Priority & Billing Type

QOS assignment determines job priority and whether usage is billed. Managed by SlurmQOS model; bootstrapped by coldfront2slurm().

Project TypeQOS AssignedBilled?
Default (PI username, no abbreviation) scavenger No — always free
Custom (with abbreviation + IO Number) normal, premium Yes — via is_billable
Per-allocation override AllocationBilling.qos FK Depends on QOS 
# Default project → free QOS
sacctmgr modify account where name=jsmith set qos=scavenger defaultqos=scavenger

# Custom project → paid QOS
sacctmgr modify account where name=jsmith_proj1 set qos=normal,premium defaultqos=normal
4. Enforcement at Job Submit

Four layers of enforcement, from user-facing to system-level:

StepWhereWhatBlocks?
1. Estimate cli_filter.lua (login node) Shows estimated cost, weekly budget remaining, prompts [y/N] No (UX only)
2. Validate cli_filter.lua Validates account exists for user Yes (rejects)
3. Accept job_submit.lua (controller) Enforces ACCEPT_COST=1 for batch jobs Yes (rejects)
4. Cap check Slurm native GrpTRESMins=billing blocks when cap reached Yes (queues/rejects)

How It Works

Weekly spend caps are enforced at the Slurm account association level using GrpTRESMins on the billing TRES. This is set directly on the account, not via per-allocation QOS.

Conversion Formula
1 SU (Service Unit) = $0.001
1 dollar = 1,000 SU
Slurm GrpTRESMins = SU-minutes

Formula: $X × 1,000 × 60 = GrpTRESMins

Example: $30/week → 30 × 1,000 × 60 = 1,800,000 SU-minutes
Example: Account-Level Caps
Project (Slurm Account)CapGrpTRESMins
arochab1_f01_a$30/weekbilling=1800000
arochab1_f01_b$20/weekbilling=1200000
bgermro1No cap(not set = unlimited)
Slurm Command
# Set a cap:
sacctmgr modify account where name=arochab1_f01_a set GrpTRESMins=billing=1800000

# Remove a cap (unlimited):
sacctmgr modify account where name=bgermro1 set GrpTRESMins=billing=-1
slurm_sync Logic
if weekly_cap_dollars:
    cap_mins = dollars_to_grptres_mins(weekly_cap_dollars)
    sacctmgr modify account where name=X set GrpTRESMins=billing={cap_mins}
# If no cap, GrpTRESMins is not set on the association (= unlimited)
Important: Weekly reset is handled by our cron task (check_and_run_core_hours_reset, daily at 1 AM), not by Slurm's native PriorityUsageResetPeriod. This allows per-allocation override schedules.
Weekly Cap Usage Sync

Every 5 minutes, the sync_weekly_cap_usage task polls Slurm via sreport for current billing-TRES usage and stores the dollar amount in the Weekly Cap Usage ($) allocation attribute.

Global Reset Schedule

Configured via CoreHoursResetSettings (singleton model). The check_and_run_core_hours_reset task runs daily at 1 AM.

PeriodReset Dates
WeeklyEvery Monday
Monthly1st of each month
QuarterlyJan 1, Apr 1, Jul 1, Oct 1
4-MonthJan 1, May 1, Sep 1
SemesterJan 1, Jul 1
AnnualJan 1
Per-Allocation Override

Set the Next Reset allocation attribute to override the global schedule for a specific allocation. When the override date is reached, only that allocation resets, and the attribute advances to the next date based on the global period.

What Happens on Reset
  • Core Usage (Hours) allocation attribute → set to 0
  • Per-user snapshots (SlurmUserUsageSnapshot) → zeroed
  • Audit log entry created
Project-Level Fields

PIs can request storage and backup when creating or updating a project:

  • storage_enabled / storage_tb — request provisioned storage (TB)
  • backup_enabled / backup_tb — request backup service (TB)

These values are copied to AllocationBilling.storage_tb / backup_tb when the allocation is activated.

Billing Rates

Storage and backup have their own rate types in the BillingRate catalog: storage (TB/month) and backup (TB/month). Configure rates in Billing Management > Rate Management.

The Terms of Service (ToS) system requires PIs to accept usage policies before creating projects or requesting allocations. Each ToS version has sub-sections that must be individually accepted.

Creating a ToS Version

There are two ways to create a ToS version:

Option A: Import from CSV (recommended)

  1. Go to Admin Tools → Import ToS in the sidebar
  2. Prepare a semicolon-delimited CSV file: "Title";"Section content..." (one row per section, no header)
  3. Enter a version number (e.g. "1.0", "2.0") and optional title
  4. Check Set as active to make it the current ToS (deactivates previous version)
  5. Upload and import

Option B: Manual via Django Admin

  1. Go to Admin → Terms of Service → Add
  2. Set version (e.g. "1.0"), title, and effective date
  3. Add sections using the inline form (Acceptable Use, Account Responsibility, Data Storage, etc.)
  4. Check Is active — this deactivates any previous version
  5. Save
Viewing Acceptance Reports
  • Admin → ToS Acceptances — shows who accepted which version, when, from which IP, and in what context (project creation, allocation request, etc.)
  • Filter by version or context to audit specific actions
Forcing Re-acceptance

Create a new ToS version (e.g. "1.1") and set it as active. All users will be required to accept the new version on their next project creation or allocation request. PIs with existing projects can also accept directly from their Project Detail page, where a warning card with ToS sections appears when acceptance is pending.

Users Who Haven't Accepted
  • Users who haven't accepted the current ToS cannot create projects or request allocations
  • The slurm_sync process also skips creating Slurm accounts for users without ToS acceptance
  • Once a user accepts, their account is created on the next sync cycle
Tip: If no ToS is active (none have is_active=True), the system allows all actions without ToS acceptance. This is the default state before any ToS is configured.

CLOACK supports three deployment models for Slurm clusters. The model is determined by the root_account field on each SlurmCluster record in Django Admin — no code changes needed.

Model A — Single Cluster (both affiliations)

root_account = "root" or "arch" (any value other than "jhu" or "schmidt").
Both JHU and Schmidt PI accounts are created under the same cluster root. The sync creates jhu and schmidt as sub-accounts under the cluster root. 

cluster_root (root/arch)
├── jhu → school → dept → PI → project
└── schmidt → ssci-* PIs
Model B — Federation (one affiliation per cluster)

root_account = "jhu" → JHU-only cluster (skips Schmidt PIs).
root_account = "schmidt" → Schmidt-only cluster (skips JHU PIs).
Multiple clusters share one slurmdbd. Users submit cross-cluster with sbatch -M cluster_name. 

slurmdbd (shared)
├── jhu-cluster (root_account=jhu) → only JHU allocations
└── schmidt-cluster (root_account=schmidt) → only Schmidt allocations
Model C — Hardware Split (both affiliations, multiple clusters)

Same as Model A but with separate clusters per hardware type (CPU, GPU). Each cluster has root_account = "arch" (or similar) and serves both affiliations. 

slurmdbd (shared)
├── arch-cpu (root_account=arch) → both JHU + Schmidt
└── arch-gpu (root_account=arch) → both JHU + Schmidt

Real-time Account Provisioning

When staff approves an allocation (status → Active), the system immediately creates the full Slurm account hierarchy via REST API: root → school → dept → PI → project account + user associations.

This is best-effort: if the REST API is unavailable, a warning is logged and the periodic sync_slurm (every 3 hours) acts as safety net. The provisioning respects Model B affiliation filtering — a JHU PI allocation approved on a Schmidt-only cluster is silently skipped.
