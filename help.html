Help & Reference Guide

Guide for PIs and users — billing, CLI tools, job submission, and more.

Before PIs can create projects and submit jobs, staff must complete these one-time setup steps:

Step 1 — Import Schools & Departments

Go to Admin Tools → Import Departments (/admin-tools/import-departments/). Upload a CSV with columns: name, code, school, school_code, affiliation (jhu or schmidt). This creates the School → Department hierarchy used for Slurm account placement and billing.

Step 2 — Create Slurm Cluster

Go to Django Admin → Slurm Clusters. Create a cluster record with:

  • Name — must match Slurm's cluster name (e.g. arch)
  • Type — docker (dev) or bare_metal (prod)
  • Root account — top of Slurm hierarchy (e.g. jhu or schmidt)
  • TRES types — e.g. cpu,billing or cpu,gres/gpu,billing
Step 3 — Create Partitions & Nodes

Go to Django Admin → Slurm Partitions. Create partition records (e.g. scavenger, premium) and assign nodes. Set AllowQos and TRESBillingWeights as needed.

Step 4 — Configure QOS

Go to Django Admin → Slurm QOS. Default records (scavenger, normal, premium) are created automatically on deploy. Mark scavenger as is_free=True so it is never billed.

Step 5 — Set Billing Rates

Go to Django Admin → Billing Rates. Create rate records for each rate type (cpu_full, cpu_om, gpu_full, gpu_om, storage, backup) with effective dates.

Step 6 — Activate Terms of Service

Go to Admin Tools → Import ToS (/admin-tools/import-tos/) to upload a ToS CSV, or create directly in Django Admin → Terms of Service. Set is_active=True on the current version. Only one version can be active at a time.

Important: Without an active ToS, all users can create projects freely. With an active ToS, users must accept all required sections before creating a project. Users who have not accepted the ToS will not get a Slurm account — even if they can log in, they cannot submit jobs.
Step 7 — Create ColdFront Resource

Go to Django Admin → Resources. Create a resource (e.g. "ARCH Cluster") that PIs will select when requesting allocations. This links ColdFront allocations to the Slurm cluster.

Ready! Once all steps are complete, PIs can create projects, request allocations, and start submitting jobs after staff approval.
  1. PI creates project — sets title, abbreviation, department, IO number, storage/backup needs. Must accept ToS if active.
  2. PI requests allocation — chooses cluster resource, enters core hours.
  3. Staff reviews & approves — allocation moves to Active.
  4. Auto-seed on activation:
    • AllocationBilling created (copies IO, department, storage/backup from project)
    • Default attributes seeded: Core Usage, Extra Core Hours, Next Reset
  5. Slurm account provisioning (real-time) — creates Slurm account hierarchy via REST API. PI must have accepted ToS for their user association to be created.
  6. ToS acceptance triggers Slurm association — when PI accepts ToS, a post_save signal immediately provisions their user association on all active allocations. If User Required is enabled on the ToS, regular users must also accept before being added.
  7. Periodic sync (safety net) — sync_slurm runs every 3 hours, catches any missed associations. Respects ToS gating.
  8. LDAP sync — ensures user exists in LDAP with home directory.
  9. Users submit jobs — billing tracked via sacct collection every 15 minutes.

Allocation Usage Gauges

Each allocation has its own Core Usage gauge. The data source depends on the allocation type:

  • Default allocations (project = PI username, no abbreviation) get scavenger QOS and are never billed.
  • Paid allocations (custom project with abbreviation) get normal/premium QOS and are billed.

Hardware Credit Pools

Hardware purchases are converted to compute credits via HardwareCreditPool. Credits offset compute charges only, not O&M. Each pool has a granted date, expiry date, and tracks credits used/remaining.

Billing Enforcement in Slurm

How ColdFront billing settings translate to Slurm enforcement via coldfront2slurm() (runs in sync_slurm scheduled task).

1. Account Hierarchy — Who Can Submit

slurm.conf enforces: AccountingStorageEnforce=associations,limits,qos. Users without a Slurm account association cannot submit jobs. 

root (jhu)
  └─ it (school)        ← fairshare from School.shares
        └─ arch (department)    ← fairshare from Department.dept_shares
              └─ jsmith (PI allocation account)
                    └─ user1, user2  ← associations
User State in ColdFrontSlurm ActionEffect
Active on allocationMaxJobs=-1 (unlimited)Can submit jobs
Removed / inactiveMaxJobs=0Cannot submit; sacct history preserved
ToS not acceptedSkipped entirelyNo Slurm association created
2. GrpTRESMins — Usage Limits

Each allocation account gets a GrpTRESMins limit derived from the Core Hours allocation attribute. The TRES types are configured per cluster via SlurmCluster.tres_types (default: cpu,billing).

Scenariocpu=billing=
No weekly cap hours × 60 hours × 60 (same value)
Weekly cap set ($X) hours × 60 (core hours) $X × 1,000 × 60 (cap in billing minutes)

Detection: when billing ≠ cpu in GrpTRESMins, a weekly cap is active.

# Example: 1000 core hours + $50 weekly cap
sacctmgr update account set grptresmins=cpu=60000,billing=3000000 where account=jsmith_proj1

# No weekly cap: billing = cpu
sacctmgr update account set grptresmins=cpu=60000,billing=60000 where account=jsmith_proj1
3. QOS — Priority & Billing Type

QOS assignment determines job priority and whether usage is billed. Managed by SlurmQOS model; bootstrapped by coldfront2slurm().

Project TypeQOS AssignedBilled?
Default (PI username, no abbreviation) scavenger No — always free
Custom (with abbreviation + IO Number) normal, premium Yes — via is_billable
Per-allocation override AllocationBilling.qos FK Depends on QOS 
# Default project → free QOS
sacctmgr modify account where name=jsmith set qos=scavenger defaultqos=scavenger

# Custom project → paid QOS
sacctmgr modify account where name=jsmith_proj1 set qos=normal,premium defaultqos=normal
4. Enforcement at Job Submit

Four layers of enforcement, from user-facing to system-level:

StepWhereWhatBlocks?

How It Works

Weekly spend caps are enforced at the Slurm account association level using GrpTRESMins on the billing TRES. This is set directly on the account, not via per-allocation QOS.

Conversion Formula
1 SU (Service Unit) = $0.001
1 dollar = 1,000 SU
Slurm GrpTRESMins = SU-minutes

Formula: $X × 1,000 × 60 = GrpTRESMins

Example: $30/week → 30 × 1,000 × 60 = 1,800,000 SU-minutes
Example: Account-Level Caps
Project (Slurm Account)CapGrpTRESMins
arochab1_f01_a$30/weekbilling=1800000
arochab1_f01_b$20/weekbilling=1200000
bgermro1No cap(not set = unlimited)
Slurm Command
# Set a cap:
sacctmgr modify account where name=arochab1_f01_a set GrpTRESMins=billing=1800000

# Remove a cap (unlimited):
sacctmgr modify account where name=bgermro1 set GrpTRESMins=billing=-1
slurm_sync Logic
if weekly_cap_dollars:
    cap_mins = dollars_to_grptres_mins(weekly_cap_dollars)
    sacctmgr modify account where name=X set GrpTRESMins=billing={cap_mins}
# If no cap, GrpTRESMins is not set on the association (= unlimited)
Important: Weekly reset is handled by our cron task (check_and_run_core_hours_reset, daily at 1 AM), not by Slurm's native PriorityUsageResetPeriod. This allows per-allocation override schedules.
Weekly Cap Usage Sync

Every 5 minutes, the sync_weekly_cap_usage task polls Slurm via sreport for current billing-TRES usage and stores the dollar amount in the Weekly Cap Usage ($) allocation attribute.

What is TRES?

Trackable Resources (TRES) are Slurm resources that can be tracked for usage and limited. Common types: cpu, mem, gres/gpu, billing.

Cluster TRES Types

Configured in Admin > Slurm Clusters > "Slurm Account Defaults" via the tres_types field. This defines which TRES types are used in GrpTRESMins limits.

Cluster Typetres_typesGrpTRESMins Example
CPU-onlycpu,billingcpu=60000,billing=60000
GPU-enabledcpu,gres/gpu,billingcpu=60000,gres/gpu=1000,billing=60000
TRESBillingWeights (per partition)

Configured on each SlurmPartition in Django Admin. Generates the TRESBillingWeights="..." directive in slurm.conf. 

# Example: 1 GPU = 2 billing units
TRESBillingWeights="CPU=1.0,Mem=0.25G,GRES/gpu=2.0"

# Default (when blank): 1 CPU-minute = 1 billing unit, all other TRES = 0

This determines how Slurm converts mixed resources into a single "billing" number. The billing TRES is what GrpTRESMins=billing=X limits against.

Per-Allocation Override

Set the Next Reset allocation attribute to override the global schedule for a specific allocation. When the override date is reached, only that allocation resets, and the attribute advances to the next date based on the global period.

What Happens on Reset
  • Core Usage (Hours) allocation attribute → set to 0
  • Per-user snapshots (SlurmUserUsageSnapshot) → zeroed
  • Audit log entry created
Project-Level Fields

PIs can request storage and backup when creating or updating a project:

  • storage_enabled / storage_tb — request provisioned storage (TB)
  • backup_enabled / backup_tb — request backup service (TB)

These values are copied to AllocationBilling.storage_tb / backup_tb when the allocation is activated.

Billing Rates

Storage and backup have their own rate types in the BillingRate catalog: storage (TB/month) and backup (TB/month). Configure rates in Billing Management > Rate Management.

Source of Truth

The Slurm account hierarchy (root → school → dept → PI → project) is driven by AllocationBilling.department, not Project.department.

Priority Order
  1. Non-default allocation billing department
  2. Default allocation billing department
  3. Project.department
  4. Fallback: SS for ssci-*, NODEP for JHU
Auto-Propagation Signals
  • Setting AllocationBilling.department on a default project → propagates to Project.department
  • Setting AllocationBilling.department → adds department to PI's UserProfile.departments
  • Adding a PI to a department → sets that department on their projects that lack one

Submit jobs programmatically (REST API)

Instead of SSH + sbatch, you can submit jobs via the ColdFront REST proxy using your personal API token. The proxy validates your allocation and forwards the job to slurmrestd with the cluster JWT — you never see the Slurm secret.

  • Generate your token on the User Profile page.
  • Full shell & Python examples (estimate + submit with cost confirmation): REST API Token — How to use.
  • Endpoints: POST /api/v1/jobs/estimate/ (dry-run) and POST /api/v1/jobs/submit/ (real submission).
  • Paid allocations require "accept_cost": true + "estimated_cost_usd" matching the server within ±5%. Free allocations (scavenger, Schmidt, no IO Number) skip the confirmation.

The Terms of Service (ToS) system requires PIs to accept usage policies before creating projects or requesting allocations. Each ToS version has sub-sections that must be individually accepted.

User Required Option

Each ToS version has a User Required checkbox (Admin → Terms of Service):

  • Disabled (default): Only PIs must accept ToS. Regular users are added to Slurm accounts without ToS acceptance.
  • Enabled: Both PIs and regular users must accept ToS before being added to Slurm accounts. Users see a ToS acceptance card on the Project Detail page.
Acceptance Flow
StepPIRegular User (when User Required)
1. Accept ToSProject creation, allocation request, or Project DetailProject Detail page
2. Signal fireson_tos_accepted → provisions Slurm user association immediately via REST API
3. Safety netsync_slurm (every 3h) catches any missed associations
Creating a ToS Version

There are two ways to create a ToS version:

Option A: Import from CSV (recommended)

  1. Go to Admin Tools → Import ToS in the sidebar
  2. Prepare a semicolon-delimited CSV file: "Title";"Section content..." (one row per section, no header)
  3. Enter a version number (e.g. "1.0", "2.0") and optional title
  4. Check Set as active to make it the current ToS (deactivates previous version)
  5. Upload and import

Option B: Manual via Django Admin

  1. Go to Admin → Terms of Service → Add
  2. Set version (e.g. "1.0"), title, and effective date
  3. Add sections using the inline form (Acceptable Use, Account Responsibility, Data Storage, etc.)
  4. Check Is active — this deactivates any previous version
  5. Optionally check User required if regular users should also accept
  6. Save
Viewing Acceptance Reports
  • Admin → ToS Acceptances — shows who accepted which version, when, from which IP, and in what context (project creation, allocation request, user onboarding, etc.)
  • Filter by version or context to audit specific actions
Forcing Re-acceptance

Create a new ToS version (e.g. "1.1") and set it as active. All PIs will be required to accept the new version on their next project creation or allocation request. PIs with existing projects can also accept directly from their Project Detail page. If User Required is enabled, regular users will also see a ToS acceptance card on the Project Detail page.

Users Who Haven't Accepted
  • PIs who haven't accepted the current ToS cannot create projects or request allocations, and their Slurm user association is not created
  • When User Required is enabled, regular users without ToS acceptance are also skipped during Slurm sync
  • Once a user accepts, a post_save signal on TOSAcceptance immediately provisions their Slurm association via REST API
  • The periodic sync_slurm (every 3 hours) acts as a safety net for any missed provisions
Tip: If no ToS is active (none have is_active=True), the system allows all actions without ToS acceptance. This is the default state before any ToS is configured.

CLOACK supports three deployment models for Slurm clusters. The model is determined by the root_account field on each SlurmCluster record in Django Admin — no code changes needed.

Model A — Single Cluster (both affiliations)

root_account = "root" or "arch" (any value other than "jhu" or "schmidt").
Both JHU and Schmidt PI accounts are created under the same cluster root. The sync creates jhu and schmidt as sub-accounts under the cluster root. 

cluster_root (root/arch)
├── jhu → school → dept → PI → project
└── schmidt → ssci-* PIs
Model B — Federation (one affiliation per cluster)

root_account = "jhu" → JHU-only cluster (skips Schmidt PIs).
root_account = "schmidt" → Schmidt-only cluster (skips JHU PIs).
Multiple clusters share one slurmdbd. Users submit cross-cluster with sbatch -M cluster_name. 

slurmdbd (shared)
├── jhu-cluster (root_account=jhu) → only JHU allocations
└── schmidt-cluster (root_account=schmidt) → only Schmidt allocations
Model C — Hardware Split (both affiliations, multiple clusters)

Same as Model A but with separate clusters per hardware type (CPU, GPU). Each cluster has root_account = "arch" (or similar) and serves both affiliations. 

slurmdbd (shared)
├── arch-cpu (root_account=arch) → both JHU + Schmidt
└── arch-gpu (root_account=arch) → both JHU + Schmidt

Real-time Account Provisioning

When staff approves an allocation (status → Active), the system immediately creates the full Slurm account hierarchy via REST API: root → school → dept → PI → project account.

User associations are gated by ToS acceptance: the PI's user association is only created if they have accepted the current ToS. Regular users are added immediately unless User Required is enabled on the ToS, in which case they must also accept first. When a user later accepts ToS, a post_save signal on TOSAcceptance triggers immediate provisioning of their Slurm association.

This is best-effort: if the REST API is unavailable, a warning is logged and the periodic sync_slurm (every 3 hours) acts as safety net. The provisioning respects Model B affiliation filtering — a JHU PI allocation approved on a Schmidt-only cluster is silently skipped.

All ColdFront → Slurm sync paths (coldfront2slurm, collect_slurm_acc_info, real-time account provisioning) use the slurmrestd REST API as the primary channel and fall back to sacctmgr CLI automatically when the REST call fails.

Why the fallback exists
  • Resilience during slurmrestd restarts, JWT rotation, or Slurm upgrades that change the REST API version.
  • Dev environments where slurmrestd is not configured still work via sacctmgr.
  • Failure is transparent: the sync run completes; the next run retries REST.
What to expect in logs

When the fallback engages you will see a single INFO line per call like:

coldfront2slurm: REST API unavailable (Connection refused), using sacctmgr CLI
collect_slurm_acc_info: REST failed (...), falling back to CLI

This is not an error — sync completed successfully via sacctmgr. Investigate only if the fallback is happening repeatedly in production.

Monitoring — Slurm Sync Metrics

Each sync call increments a counter per (cluster, hour, path, operation). Open Django Admin → Slurm Sync Metrics to see, for each hour bucket:

Pathrest (REST succeeded) or cli (fell back to sacctmgr)
OperationWhich sync function: coldfront2slurm, collect_slurm_acc_info, …
CountNumber of times that path ran in the hour
Last errorMost recent REST failure reason (only populated for cli rows)

Healthy cluster: rest rows accumulate, cli rows are absent or rare. When the hourly cli count reaches 5 a WARNING log line is emitted once per bucket so it shows up in Service Logs without flooding.

Forcing a specific path (debugging only)
  • Force CLI: stop slurmrestd or set an invalid slurmrestd_url on the cluster; all operations will fall back to sacctmgr.
  • Force REST: there is no override — REST is always tried first. If it fails the CLI fallback will run. Unit tests (test_sync_metrics) exercise both branches explicitly.

Both dev and prod Slurm clusters run Slurm 25.11.2 with JWT authentication enabled on slurmrestd. This is what allows ColdFront → Slurm REST calls (the fast path in Slurm Sync: REST API + CLI Fallback).

How it is wired
  • slurm.conf → AuthAltTypes=auth/jwt, AuthAltParameters=jwt_key=/var/spool/slurm/jwt_hs256.key
  • slurmrestd started with -a local,jwt (accepts UNIX-socket local auth and TCP JWT)
  • Token file: ${BASE_ETC}/slurm/slurm_jwt.token on the host, bind-mounted into coldfront and slurmrestd at /etc/slurm/slurm_jwt.token
  • Renewed daily at 03:00 via crontab (30-day lifespan, scontrol token lifespan=2592000)
  • ./start.sh slurm start and slurm restart regenerate the token on the way up (5 retries × 2 s backoff)
Symptom: REST falls back to sacctmgr CLI

If Slurm Sync Metrics show persistent cli rows and the logs say "UNIX socket not available and no JWT token configured", the token file is likely empty or stale.

  1. Check on host: wc -c base/slurm/slurm_jwt.token (should be ~140+ bytes, not 0).
  2. Check in container: docker exec coldfront wc -c /etc/slurm/slurm_jwt.token. If 0 despite the host having content, it's the virtiofs cache pitfall — restart coldfront: docker restart coldfront.
  3. Regenerate manually: 
    docker exec arch-dev-slurmctld scontrol token lifespan=2592000 \
  | grep SLURM_JWT | cut -d= -f2 > base/slurm/slurm_jwt.token
docker restart coldfront
  4. Smoke-test from inside coldfront: 
    docker exec coldfront python -c "
from coldfront.plugins.arch_sync.models import SlurmCluster
from coldfront.plugins.arch_sync.slurm_client import SlurmClient
c = SlurmCluster.objects.first()
SlurmClient.from_cluster(c).ping()
print('REST ping OK')
"
Why the CLI fallback still matters

JWT tokens expire (30 days). The cron job at 03:00 refreshes them, but if that fails, the next sync run should still succeed via sacctmgr. Watch Slurm Sync Metrics — a sudden spike in cli rows is the signal to regenerate the token.

ARCH clusters expose Slurm partitions by purpose and QoS by use case. Pick the partition that matches your workload, and the QoS that matches the priority you need.

Partition naming convention
  • CPU partitions by billing tier: low, med, high — differ in TRES billing weight (cost per core-hour).
  • GPU partitions by hardware: a100, l40s, h100, h200 — submit to the partition matching the GPU type your job needs.
QoS
  • scavenger — free, preemptible. Long max wall, no cost. Your job may be killed if a paid job needs the resources.
  • interactive — higher priority, short max wall (4h), capped resources per user. Use for tuning, debugging, or working in a notebook.
  • normal/premium — standard paid QoS for batch work (where enabled by your allocation).
Example submissions
sbatch --partition=med --qos=normal job.sh         # paid CPU work
sbatch --partition=scavenger --qos=scavenger run.sh  # free, preemptible
srun  --partition=a100 --qos=interactive --pty bash  # 4h interactive on A100

Partitions and QoS are seeded per SlurmCluster and managed in Django Admin. The same naming convention is used for all clusters so users can submit jobs the same way regardless of cluster.

Convention
  • CPU partitions: low / med / high — one per billing tier. Tier code on BillingRate joins to the partition.
  • GPU partitions: named after the hardware (a100, l40s, h100, h200). Set kind=gpu and hardware=<model> on the partition.
  • QoS: scavenger (free, preemptible) and interactive (priority=1000, max_wall=04:00:00, resource cap) seeded on every cluster. normal/premium remain for backward compat.
Seeding (idempotent)
./start.sh populate dev --partitions a100,l40s,h100,h200,low,med,high,scavenger

--partitions takes a comma-separated list of partition names (no preset). QoS records (scavenger, interactive, normal, premium) are bootstrapped automatically by slurm_sync from the SlurmQOS model — no --qos flag is required.

Multi-cluster

Each SlurmCluster gets its own copy of the partition/QoS set. Run the populate command after creating a new cluster, or use the admin to add per-cluster. slurm_sync writes the resulting slurm.conf and bootstraps QoS in slurmdbd.

End-to-end billing pipeline. Each link in the chain is editable from Django Admin.

1. BillingRate — rate catalog

/admin/arch_sync/billingrate/

  • cluster / partition_name — scope (blank = all clusters / all partitions)
  • rate_type — cpu_full, cpu_om, gpu_full, gpu_om, storage, backup
  • rate_per_unit — $ per SU (CPU), GPU-hour, or TB-month
  • effective_from / effective_to — date-versioned. Leave effective_to blank for active rate.

Two-track pricing: *_full for pay-per-use PIs (compute + O&M bundled). *_om for hardware-credit holders (lower — O&M only).

2. AllocationBilling — per-allocation config (one per Allocation)

/admin/arch_sync/allocationbilling/ — auto-created by signal _ensure_allocation_billing on allocation save.

  • io_number — JHU service-center cost code. No IO → is_billable=False automatically.
  • department — drives the Slurm account hierarchy (root → school → dept → PI → project)
  • billing_type — pay_per_use / credit_holder / no_charge
  • is_billable — single source of truth for whether the allocation is charged. False = tracked but $0.
  • storage_tb / backup_tb — provisioned TB (multiplied by storage/backup rate per month)
  • qos — FK to SlurmQOS; overrides cluster default (e.g. interactive)

Auto-flags is_billable=False + billing_type=no_charge for: default allocations, Schmidt PIs (ssci-*), allocations without IO Number.

3. HardwareCreditPool — prepaid credits from hardware purchase

/admin/arch_sync/hardwarecreditpool/

  • allocation — FK to the Allocation receiving the credits
  • hardware_value — dollar value of the hardware
  • total_credits — compute hours granted (e.g. 87600 GPU-hours)
  • credit_type — cpu, gpu, storage, memory (matches BillingRate.rate_type prefix)
  • granted_date / expiry_date — validity window

Credits cover compute only. O&M is always billed in cash via *_om rates.

4. SlurmPartition.tres_billing_weights — per-partition multiplier

/admin/arch_sync/slurmpartition/

Injected as TRESBillingWeights= in slurm.conf. Example: CPU=1.0,Mem=0.25G,GRES/gpu=2.0 → 1 GPU-hour = 2 billing units. Combined with BillingRate at sreport time.

5. SlurmAccountUsagePeriod — monthly usage aggregate

/billing/report/ · /billing/my-usage/

Auto-generated by collect_billing (cron, daily). Drives invoices and the PI Usage & Billing dashboard.

Staff Help (In-App Reference)

The following sections are available to every helpdesk staff user at /extensions/staff-help/. They summarise the Standard Operating Procedures (SOPs) enforced by the helpdesk automation.

1. First Quality Response (FQR)

FQR is the quality standard for every first interaction. A good FQR is prompt, complete, and effective — addressing the user’s concern meaningfully and moving toward resolution on first contact.

Pre-Work Checklist (before responding):

  • Research the issue — review error codes, logs, documentation
  • Know the user — check RT history, environment, previous cases, severity/SLA
  • Know the offerings — scientific software vs system, support boundaries, mission-critical
  • Identify resources — workflows, KB articles, SMEs, swarming guidance

Core Elements of an FQR:

  • Clearly Defined Scope — ask clarifying questions
  • Progress Toward Resolution — move it forward on first contact
  • Empathy & Human Communication — no robotic phrasing
  • Alignment on Urgency — match the user’s sense of priority
  • Minimal User Effort — reduce back-and-forth
  • Timely Response — meet SLA while preparing
  • Leadership and Guidance — communicate the resolution plan
2. Escalation Chain

Tickets escalate automatically along the tier chain when aged beyond the policy’s threshold. SSEs can also escalate manually by reassigning. When escalated, the previous assignee becomes a collaborator (role escalated_from) to preserve context.

4. 3-Strike Follow-Up Process

When a user doesn’t reply to a staff message, the system sends automatic follow-up emails at fixed business-day intervals. A user reply at any point resets the strike counter.

  1. Strike 1 (after 2 business days) — gentle reminder
  2. Strike 2 (3 business days after Strike 1) — checking in, encouraging engagement
  3. Strike 3 (3 business days after Strike 2) — case archiving notice; ticket closes
5. Response Templates

Default templates are seeded and available from the ticket composer. Placeholders: {name}, {your_name}, {ticket_number}, {issue}, {signature}, {survey_url}.

  • FQR — Standard First Response: introduction, scoping questions, user survey link
  • FQR — SLA Escalation: triage + hand-off to Systems Team via rt-triage
  • Strike 1 — Standard Follow-Up
  • Strike 1 — Tutorial Follow-Up (after sending tutorial content)
  • Strike 2 — Encouraging Engagement
  • Strike 3 — Case Archiving Notice
